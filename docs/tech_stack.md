## Justification by Layer

### Frontend: React + Next.js

**Why React?**
- Largest ecosystem; extensive component libraries for tables and forms
- Strong TypeScript integration
- React Query provides excellent server state management (cached API data)

**Why Next.js?**
- App Router supports SSR for faster initial load (dashboard loads with data)
- File-based routing reduces boilerplate
- Vercel deployment is seamless with zero-config

**Alternatives considered:**
- **Vite + React SPA**: Simpler, but loses SSR benefits for initial data-heavy dashboard
- **Vue 3 + Nuxt**: Comparable option, but React ecosystem is stronger for enterprise tooling

---

### Backend: Node.js + Express + TypeScript

**Why Node.js?**
- Fast async I/O well-suited for high-concurrency API handling
- Shared TypeScript types between frontend and backend possible
- Large ecosystem of libraries for JWT, validation, ORM

**Why Express?**
- Minimal, flexible, widely understood
- Easy to layer middleware for auth, validation, rate limiting
- Easy to migrate routes to independent microservices later

**Why TypeScript (strict)?**
- Type safety prevents entire classes of runtime bugs
- Enforces correct data shapes from DB through to API response
- Excellent IDE support for team productivity

**Alternatives considered:**
- **Laravel (PHP)**: Strong option, especially with existing team familiarity. Choose this if team is PHP-first.
- **FastAPI (Python)**: Great for AI/ML integration heavy systems; overkill here.
- **NestJS**: More opinionated, better for larger teams; Express is leaner for this scope.

---

### Database: PostgreSQL

**Why PostgreSQL?**
- Full ACID compliance essential for stock integrity (atomic transactions)
- Rich indexing (BTREE, composite, partial)
- Row-level locking for concurrent operation validation
- Native UUID support
- Excellent JSON support for future flexible data
- Triggers for ledger immutability enforcement

**Alternatives considered:**
- **MySQL**: Acceptable, but Postgres has better ACID guarantees and richer feature set
- **MongoDB**: Document DB is wrong fit for relational inventory data with strict integrity needs
- **SQLite**: Only for development; not for production multi-user concurrency

---

### ORM: Knex.js (or Prisma)

**Why Knex.js?**
- Query builder (not full ORM) gives full SQL control
- Easy parameterized queries — no magic
- Good for complex joins like ledger aggregates

**Why Prisma as alternative?**
- Type-safe queries autogenerated from schema
- Better DX for teams that prefer schema-first approach
- Slightly more opinionated but faster to write

**Recommendation:** Prisma for teams new to the codebase; Knex for maximum query control.

---

### Cache: Redis

**Why Redis?**
- Sub-millisecond read/write for OTP and KPI caching
- TTL-based key expiry built-in — perfect for OTP (15 min) and sessions (7 days)
- Atomic operations prevent race conditions in rate limiting
- Widely supported managed services (Upstash, Redis Cloud, AWS ElastiCache)

**Alternatives considered:**
- **Memcached**: Simpler but lacks TTL and persistence features
- **In-memory (Node.js)**: Not suitable for multi-instance deployments

---

### Authentication: JWT + bcrypt + Redis

**Why JWT?**
- Stateless — no DB lookup on every request
- Works across multiple API instances without shared session DB
- Short TTL (15 min) limits exposure window

**Why bcrypt?**
- Intentionally slow — resistant to brute force and GPU cracking
- Industry standard for password hashing
- Cost factor 12 gives ~200ms per hash — acceptable for login

**Why Redis for refresh tokens?**
- Instant revocation on logout (DELETE from Redis)
- Pure DB token storage requires index lookup on every refresh

---

### Email: SendGrid

**Why SendGrid?**
- Reliable delivery for OTP emails — critical for auth flow
- Good deliverability rate and spam handling
- Simple REST API with Node.js SDK
- Generous free tier for MVP

**Alternatives considered:**
- **AWS SES**: Better at scale, but more configuration overhead at MVP stage
- **Mailgun**: Comparable option — acceptable substitute
- **SMTP self-hosted (Postfix)**: Unreliable deliverability; not recommended for OTPs

---

### DevOps: Docker + GitHub Actions + Railway/Vercel

**Why Docker?**
- Reproducible environments across dev, staging, and production
- Simplifies dependency management
- Language for deployment to any cloud (Railway, AWS ECS, GCP Cloud Run)

**Why GitHub Actions?**
- Native GitHub integration
- Extensive marketplace actions (Docker, security scans, Vercel deploy)
- Free for open source; reasonable pricing for private repos

**Why Railway/Render for MVP?**
- Zero-ops deployment — no server management, no Kubernetes
- Automatic SSL, custom domains, environment variables UI
- PostgreSQL and Redis add-ons available
- Scales to v2.0 needs; migrate to AWS/GCP only when operationally justified

## Technology Decision Log

| Decision | Option Chosen | Option Rejected | Reason |
|---|---|---|---|
| API framework | Express | NestJS | Simpler, less boilerplate for this scope |
| Database | PostgreSQL | MongoDB | ACID required for stock integrity |
| Cache | Redis | In-memory | Multi-instance support needed |
| Auth | JWT | Session cookies | Stateless, multi-instance friendly |
| ORM | Prisma | Sequelize | Type safety, schema-first, better DX |
| Hosting API | Railway | AWS ECS | Lower ops overhead at MVP |
| CI/CD | GitHub Actions | CircleCI | Native GitHub integration |
